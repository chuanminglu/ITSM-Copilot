## 1️⃣ 5 Whys根因分析

### 痛点P1：业务部门经常选错系统

**问题描述**: 业务部门在提新需求时要选择相关系统，因为系统很多（30+个），经常选错系统，导致IT支持部门分派需求时产生困扰。

**5 Whys分析链**：

```
├── Why1：为什么会经常选错系统？
│   → 因为系统太多（30+个），业务人员不清楚各系统的功能归属边界
│
├── Why2：为什么不清楚各系统的功能归属？
│   → 因为业务人员只了解业务需求和流程，不了解IT系统架构和技术实现
│
├── Why3：为什么业务人员不了解IT系统架构？
│   → 因为缺少"业务场景→系统功能"的清晰映射文档和引导工具
│
├── Why4：为什么缺少这样的映射和引导？
│   → 因为系统文档分散在各处，没有统一入口，且文档是技术语言而非业务语言
│
└── Why5：根本原因是什么？
    → 🎯 **根因：缺少基于需求内容的智能系统推荐能力，无法将业务语言自动翻译成系统选择**
```

**解决方向**：

1. 构建"需求关键词→系统"的智能推荐引擎
2. 提供系统知识库（用业务语言描述系统职责）
3. 降低选择认知负担（分类、搜索、向导式引导）
4. 提供兜底方案（不确定时可寻求帮助）

**根因验证**：

- ✅ 如果有智能推荐，业务人员输入需求后自动推荐系统 → 选错率降低
- ✅ 如果有系统知识库，业务人员可查看系统职责 → 理解系统边界
- ✅ 如果有分类和搜索，业务人员可快速定位 → 减少盲选

---

### 痛点P2：业务部门需求描述太简单

**问题描述**: 业务部门需求目标与需求描述太简单，可能只有一句话需求，导致接收需求的部门在需求澄清与需求理解方面花费大量的时间。

**5 Whys分析链**：

```
├── Why1：为什么需求描述只有一句话？
│   → 因为业务人员不知道需求应该包含哪些信息（背景、目标、场景、验收标准等）
│
├── Why2：为什么不知道应该包含哪些信息？
│   → 因为业务人员缺乏需求工程知识，没有接受过"如何写好需求"的培训
│
├── Why3：为什么缺乏需求工程知识？
│   → 因为业务人员的工作重心是业务运营，而非需求编写，缺少结构化思维框架
│
├── Why4：为什么没有结构化思维框架？
│   → 因为ITSM系统只提供自由文本框，没有任何引导和提示，全靠用户自觉
│
└── Why5：根本原因是什么？
    → 🎯 **根因：缺少需求提交的引导式脚手架（模板/检测/提示），用户处于"不知道自己不知道什么"的状态**
```

**解决方向**：

1. 提供结构化需求模板（功能优化/新功能/Bug修复等）
2. 实时检测需求完整性，提示缺失信息
3. 使用引导式对话或问题生成器，逐步补充关键信息
4. 提供高质量需求示例，降低描述门槛（"抄作业"）
5. 边用边学，嵌入需求编写技巧教育

**根因验证**：

- ✅ 如果有模板引导，业务人员知道该填什么 → 需求完整度提升
- ✅ 如果有实时检测，业务人员在提交前就知道缺什么 → 减少澄清轮次
- ✅ 如果有对话引导，业务人员像聊天一样补充信息 → 降低填表抵触
- ✅ 如果有示例参考，业务人员可以模仿 → 降低思考成本

## 3️⃣ ICE评分矩阵

### 评分维度说明

**Impact（影响力）** (1-10分)：对业务价值/用户体验的影响程度

- 10分：直接解决核心痛点，影响100%用户，显著提升效率（如需求澄清轮次减少50%）
- 7分：解决重要问题，影响大部分用户，明显改善体验
- 4分：有价值但非核心，影响部分用户
- 1分：锦上添花，影响很小

**Confidence（信心）** (1-10分)：团队对方案成功的信心程度

- 10分：技术成熟、团队熟悉、风险极低（如纯前端规则引擎）
- 7分：技术可行、有类似经验、风险可控
- 4分：技术不确定性较高，需要调研验证
- 1分：技术风险高，成功概率低

**Ease（易实现性）** (1-10分)：技术实现难度和资源需求（10=最容易）

- 10分：1-2天可完成，技术简单，无依赖（如UI优化）
- 7分：3-5天可完成，技术难度中等
- 4分：6-10天可完成，技术复杂度较高
- 1分：超过10天，需要大量资源

**ICE得分** = (Impact + Confidence + Ease) / 3

## 4️⃣ MVP功能范围定义

### 资源约束计算

```
可用资源 = 2人 × 10工作日 = 20人天
Buffer预留 = 20%（应对风险和调试）
实际可分配 = 20 × 0.8 = 16人天
```

## 5️⃣ 决策依据与风险提示

### 决策依据

#### 1. P0功能选择理由

**选择T4+T3+T7组合的理由**：

✅ **根因匹配度最高**

- T4和T3直接解决P2根因（缺少引导式脚手架）
- T7直接解决P1根因（缺少智能推荐和引导）
- 三者组合覆盖5 Whys分析得出的所有根因

✅ **ICE得分最高**

- T4得分9.0（Impact 9 + Confidence 9 + Ease 9）
- T7得分9.0（Impact 8 + Confidence 10 + Ease 9）
- T3得分8.7（Impact 9 + Confidence 9 + Ease 8）
- 平均得分8.9，远超其他方案

✅ **技术可行性高**

- 完全前端化，无需后端服务（满足技术约束）
- 基于规则引擎和模板，无需复杂AI（降低风险）
- 团队有前端开发经验，技术成熟可靠

✅ **快速见效**

- 实时检测和模板引导，上线即见效
- 预防式而非事后补救，立即改善用户体验
- 8天可完成核心功能，符合10工作日约束

✅ **可持续优化**

- 检测规则可持续优化（基于真实使用数据）
- 模板可持续扩充（根据用户反馈）
- 为未来AI能力升级预留空间

#### 2. P1功能延迟理由

**为什么T1智能推荐（ICE=7.3）不在P0**：

⚠️ **需要数据准备**

- 需要导出和清洗历史需求数据（1-2天）
- 需要构建系统-关键词映射规则（人工标注）
- 数据质量直接影响推荐准确率

⚠️ **技术不确定性**

- 基础版（关键词匹配）准确率可能不够高
- 智能版（AI语义理解）需要API或前端LLM
- 调试和优化耗时，可能超出时间约束

✅ **有替代方案**

- T7的系统分类和搜索可部分解决P1
- 用户可先通过分类和搜索找到系统
- 智能推荐作为增强功能，后续迭代

**为什么T2对话式引导（ICE=6.0）在P2**：

⚠️ **开发周期长**

- 完整对话式体验需10-15天
- 多轮对话状态管理复杂
- 提示词设计和调优耗时

⚠️ **技术复杂度高**

- 需要AI能力（前端LLM或API调用）
- 技术选型需要调研验证
- 风险较高，可能影响交付

✅ **有替代方案**

- T3模板引导和T4实时检测可实现类似效果
- 引导式填写+智能提示=轻量版对话
- 对话式作为未来方向，先验证需求

#### 3. 资源分配合理性

**工时分配**：

- P0核心功能：15.5人天（77.5%）
- 集成测试部署：4人天（20%）

- 如果进度超前，可增加P1功能
- 如果进度延迟，可削减部分P0功能的完善度
