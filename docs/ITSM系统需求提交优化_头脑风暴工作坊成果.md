# ITSM系统需求提交优化 - 头脑风暴工作坊成果

## 1️⃣ 工作坊准备

### HMW 问题回顾

基于需求分析阶段的成果（来自需求分析），本次头脑风暴围绕以下 HMW 问题展开：

**HMW-1**: 我们如何帮助**业务人员**在**提交需求时**准确选择相关系统，**即使他们不了解系统架构**？

**HMW-2**: 我们如何帮助**业务人员**在**描述需求时**提供完整信息，**而不让他们觉得是在填写复杂表单**？

**HMW-3**: 我们如何让**需求提交过程**成为一个**引导式对话**，而非**填表任务**？

### 核心洞察回顾

> 💡 **核心洞察**（来自需求分析）:
> 
> 这不是一个"用户偷懒"的问题，而是**认知鸿沟**问题。业务人员处于"不知道自己不知道什么"的状态。他们缺乏需求工程知识框架，不知道如何将业务问题翻译成技术语言。
> 
> **真正需要的是**：一个"智能翻译官" + "思维脚手架"，帮助业务人员**在不增加负担的前提下**，自然而然地提供完整信息。

### 技术约束与时间约束

**技术约束**:
- ✅ 使用浏览器插件实现
- ✅ 尽量不要有后端服务

**时间约束**:
- ⏰ 2个人在10个工作日完成
- ⏰ 技术选型需要简单

### 引导问题

为了激发创意思维，我们设计了以下引导问题：

1. 🤔 **用户视角**: 如果你是业务人员，提需求时最困扰的是什么？
2. 🤖 **AI助手视角**: 如果有一个 AI 助手帮你提需求，你希望它能做什么？
3. 📋 **IT视角**: IT 支持人员在接收需求时，最希望需求包含哪些信息？
4. 🚀 **理想视角**: 如果没有任何技术限制和时间限制，理想的需求提交体验是怎样的？
5. 💡 **借鉴视角**: 其他领域（如电商下单、医院挂号、客服咨询）的类似问题，是怎么解决的？
6. ⚡ **快速验证**: 有什么方法可以在1-2天内快速验证想法的有效性？

### 头脑风暴规则

> 🎯 **核心原则**: 数量优先，延迟评判

| 规则         | 说明                                               |
| ------------ | -------------------------------------------------- |
| ❌ 禁止批评 | 发散阶段不评价任何想法的可行性，不说"这不行"       |
| ✅ 鼓励疯狂 | 越不寻常的想法越有价值，即使看似不可能实现         |
| 🔗 借力发挥 | 在他人想法上叠加和改进："Yes, and..."              |
| 📊 数量优先 | 目标：生成至少30个想法，每个维度至少5个            |
| 🎨 可视化   | 用简洁的语言描述想法，便于理解和引用               |
| ⏱️ 时间盒 | 发散阶段专注创意，收敛阶段考虑约束（10工作日限制） |

### 热身活动

**"100种用途"练习** (3分钟)

> 💭 **问题**: 一个回形针有哪些非常规用途？
> 
> **目的**: 打开思维，进入创意状态，突破"常规答案"的思维定式。
> 
> **示例答案**: 书签、临时拉链、清理键盘缝隙、做手机卡槽针、首饰、钓鱼钩、艺术品...

## 2️⃣ 发散阶段 - 创意生成

### 围绕 HMW-1: 系统选择问题（解决痛点1）

💡 **想法 #1**: 🔧 AI语义理解系统推荐引擎
   📝 **详细说明**: 用户输入需求描述后，浏览器插件实时调用前端AI模型（如WebLLM或Chrome内置AI）分析语义，自动推荐最匹配的Top 3系统，并显示推荐理由（如"检测到关键词：计费、账单"）。
   🎯 **解决的痛点**: P1 - 业务人员不知道选哪个系统
   🌟 **创新点**: 从"用户选系统"变为"系统找用户"，降低选择负担

💡 **想法 #2**: 🔧 基于历史数据的关键词匹配
   📝 **详细说明**: 分析历史需求数据，提取"需求关键词→系统"的映射规则（如"计费"→计费系统），打包在插件中。用户输入时实时匹配关键词，推荐系统。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 利用历史智慧，无需复杂AI模型，完全前端化

💡 **想法 #3**: 🎨 系统选择向导（决策树）
   📝 **详细说明**: 插件弹出3-5个简单问题（如"这个需求跟客户有关吗？""涉及计费吗？"），根据回答引导用户选择系统，像装机向导一样。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 降低选择的认知负担，用问答代替下拉框

💡 **想法 #4**: 🔧 系统智能搜索（模糊匹配）
   📝 **详细说明**: 将系统下拉框改造为搜索框，用户输入关键词（如"客户"），插件自动过滤和推荐相关系统，支持拼音、模糊匹配。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 化主动选择为主动搜索，更符合用户习惯

💡 **想法 #5**: 📋 相似需求案例推荐
   📝 **详细说明**: 用户输入需求描述后，插件搜索历史相似需求（基于向量相似度或关键词匹配），展示"其他人的类似需求选择了XX系统"。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 用户可以"抄作业"，参考他人经验

💡 **想法 #6**: 🎓 系统知识卡片（即时学习）
   📝 **详细说明**: 当用户悬停在某个系统名称上时，插件弹出卡片，显示该系统的职责、常见场景、示例需求，帮助用户理解和选择。
   🎯 **解决的痛点**: P1 - 认知负载过高
   🌟 **创新点**: 边用边学，提升用户系统架构认知

💡 **想法 #7**: 🔧 系统分类和标签化
   📝 **详细说明**: 将几十个系统按业务领域分类（客户类、计费类、网络类等），用户先选分类再选系统，降低一次性展示的系统数量。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 分层选择，减少认知负载

💡 **想法 #8**: 🤝 错误分派反馈学习
   📝 **详细说明**: 当IT人员重新分派需求时，记录"用户选了X系统，实际应该是Y系统"的数据，反馈给插件优化推荐模型。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 持续学习优化，准确率越来越高

💡 **想法 #9**: 🎨 可视化系统地图
   📝 **详细说明**: 将系统架构以可视化地图形式展示（如思维导图或关系图），用户点击业务场景自动关联到系统。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 可视化降低理解门槛

💡 **想法 #10**: 🎓 系统选择教程（新手引导）
   📝 **详细说明**: 首次使用时，插件提供30秒互动教程，展示如何根据需求类型选择系统，提升用户能力。
   🎯 **解决的痛点**: P1 - 需求工程能力缺失
   🌟 **创新点**: 授人以渔，长期提升用户能力

💡 **想法 #11**: 🔧 系统默认值智能预填充
   📝 **详细说明**: 基于用户的部门、角色、历史行为，预填充最可能的系统选项（如销售部门默认推荐CRM系统）。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 个性化推荐，减少手动操作

💡 **想法 #12**: 📋 系统负责人联系方式提示
   📝 **详细说明**: 当用户选择某个系统时，显示该系统负责人的联系方式，"不确定选这个？可以先咨询XX（电话/邮箱）"。
   🎯 **解决的痛点**: P1 - 选择焦虑
   🌟 **创新点**: 提供兜底方案，降低选错的心理压力

💡 **想法 #13**: 🔧 多系统关联推荐
   📝 **详细说明**: 当需求可能涉及多个系统时，插件提示"这个需求可能同时需要XX和YY系统配合"，支持多选。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 处理复杂场景，避免遗漏

💡 **想法 #14**: 🎨 系统选择信心度反馈
   📝 **详细说明**: 用户选择系统后，要求用户评价信心度（确定/不太确定/不确定），信心度低时IT部门优先核实，减少错误分派成本。
   🎯 **解决的痛点**: P1
   🌟 **创新点**: 显性化用户的不确定性，帮助IT部门优先处理

💡 **想法 #15**: 🔧 临时"不知道选哪个"选项
   📝 **详细说明**: 增加一个"不确定，请IT部门帮我选择"的选项，用户选择后必须填写详细需求描述，由IT部门分派（明确告知会增加处理时间）。
   🎯 **解决的痛点**: P1 - 选择焦虑
   🌟 **创新点**: 提供兜底方案，但有意识引导用户尝试自己选择

### 围绕 HMW-2: 需求描述问题（解决痛点2）

💡 **想法 #16**: 📋 智能需求模板推荐
   📝 **详细说明**: 根据用户选择的系统或输入的关键词，自动推荐合适的需求模板（功能优化类/新功能开发类/Bug修复类/数据需求类/权限申请类），每个模板包含结构化字段。
   🎯 **解决的痛点**: P2 - 需求描述只有一句话
   🌟 **创新点**: 从自由文本到结构化引导，降低描述门槛

💡 **想法 #17**: 💬 AI对话式需求澄清助手
   📝 **详细说明**: 用户输入一句话需求后，AI以对话形式追问关键信息（如"这个功能主要给谁用？""预期达到什么效果？"），边聊边补充需求。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 像聊天一样提需求，降低填表压力

💡 **想法 #18**: 🔧 需求完整性实时检测
   📝 **详细说明**: 插件实时分析需求描述，高亮缺失的关键信息（如未提及使用场景、未说明预期效果），并提示"建议补充XX信息"。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 实时反馈，预防问题而非事后补救

💡 **想法 #19**: 📋 5W2H引导式填写
   📝 **详细说明**: 将需求描述框改造为引导式问答：Who（谁用）、What（做什么）、When（什么时候）、Where（哪里）、Why（为什么）、How（怎么做）、How much（成本/规模），用户逐项填写。
   🎯 **解决的痛点**: P2 - 需求工程能力缺失
   🌟 **创新点**: 提供思维脚手架，确保完整性

💡 **想法 #20**: 💬 智能问题生成器
   📝 **详细说明**: 基于用户的一句话需求，AI生成3-5个追问（如"影响多少用户？""现在怎么处理这个问题？"），用户点击问题即可在描述框插入提示文本。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 降低思考成本，引导而非强制

💡 **想法 #21**: 📋 需求示例库参考
   📝 **详细说明**: 根据用户选择的需求类型，展示2-3个高质量需求示例，"其他人是这样描述的"，用户可以参考补充。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 用户可以"抄作业"，降低描述门槛

💡 **想法 #22**: 🔍 需求质量评分（实时反馈）
   📝 **详细说明**: 插件实时评估需求描述质量，显示评分（如60分 - 信息不足 / 85分 - 良好 / 95分 - 优秀），并提示如何提升到更高分数。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 游戏化反馈，激发用户完善需求的动力

💡 **想法 #23**: 💬 语音转文字 + AI优化
   📝 **详细说明**: 用户用语音描述需求（30秒-1分钟），插件自动转文字并通过AI优化成结构化需求描述，用户确认后提交。
   🎯 **解决的痛点**: P2 - 填表抵触
   🌟 **创新点**: 降低输入门槛，语音比打字更自然

💡 **想法 #24**: 🔍 关键信息缺失警告
   📝 **详细说明**: 当用户点击提交时，插件检测必要信息（如背景、目标、验收标准），如果缺失弹出警告："还缺少XX信息，IT部门可能需要联系你澄清，是否继续提交？"
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 在提交前拦截，给用户最后一次机会

💡 **想法 #25**: 📋 需求字数和结构提示
   📝 **详细说明**: 显示实时字数统计，并提示"建议至少150字，包含背景、目标、场景"，当达到标准时显示✅。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 明确标准，降低不确定性

💡 **想法 #26**: 🔧 AI需求扩写助手
   📝 **详细说明**: 用户输入一句话需求后，点击"AI扩写"，插件基于常见模式生成完整版需求描述（如补充背景、使用场景、预期效果），用户修改后提交。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: AI代劳，用户只需审核和修改

💡 **想法 #27**: 🎓 需求编写技巧卡片
   📝 **详细说明**: 在需求描述框旁边显示"💡提示"按钮,点击后展示需求编写技巧（如"好的需求包含：现状问题 + 期望效果 + 使用场景"）。
   🎯 **解决的痛点**: P2 - 需求工程能力缺失
   🌟 **创新点**: 即时教育，边用边学

💡 **想法 #28**: 📋 需求草稿自动保存
   📝 **详细说明**: 插件自动保存用户输入的需求草稿（存储在浏览器LocalStorage），下次打开时可以继续编辑，避免信息丢失。
   🎯 **解决的痛点**: P2 - 时间压力
   🌟 **创新点**: 降低一次性完成压力，支持分批填写

💡 **想法 #29**: 🤝 需求协作编辑（同事帮忙）
   📝 **详细说明**: 生成需求草稿分享链接，业务人员可以邀请同事一起完善需求描述，协作编辑后再提交。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 利用团队智慧，降低个人负担

💡 **想法 #30**: 🔧 需求附件智能提取
   📝 **详细说明**: 用户上传附件（如Word文档、截图、流程图）时，插件自动提取关键信息（如OCR识别截图文字）补充到需求描述中。
   🎯 **解决的痛点**: P2
   🌟 **创新点**: 多模态信息融合，减少手动输入

### 💎 疯狂但有启发的想法

这些想法可能短期内无法实现，但能激发新的思路：

🚀 **想法 #31**: 需求预测系统（主动式）
   📝 **详细说明**: 基于业务部门的历史需求模式和当前业务数据，AI主动预测可能需要的需求（如"检测到客户投诉率上升，是否需要优化客户服务系统？"），用户确认后快速提交。
   🌟 **启发点**: 能否从"被动接收"变为"主动预测"？是否可以用业务指标触发需求建议？

🚀 **想法 #32**: 需求众包评审（协作式）
   📝 **详细说明**: 业务人员提交需求后,系统自动推送给其他相关业务人员和IT人员，大家可以点评、补充、投票，众包完善需求质量。
   🌟 **启发点**: 能否利用群体智慧？需求完善是否可以社区化？

🚀 **想法 #33**: 需求自动生成（极简式）
   📝 **详细说明**: 用户只需回答3个超级简单的问题（遇到什么问题？希望怎么样？多重要？），AI自动生成完整的结构化需求文档。
   🌟 **启发点**: 能否将用户输入降到最低？AI能否真正理解意图？

🚀 **想法 #34**: 需求知识图谱（关联式）
   📝 **详细说明**: 构建需求知识图谱，当用户提新需求时，自动关联历史相似需求、解决方案、经验教训，"上次XX部门提过类似需求，最终方案是YY"。
   🌟 **启发点**: 能否让需求成为知识资产？历史经验如何复用？

🚀 **想法 #35**: 需求游戏化（激励式）
   📝 **详细说明**: 提交高质量需求获得积分，排行榜展示"需求之星"，积分可兑换奖励，提升业务人员主动完善需求的动力。
   🌟 **启发点**: 能否用游戏化机制改变行为？如何让"好好提需求"成为荣誉？

🚀 **想法 #36**: 需求视频录制（多模态）
   📝 **详细说明**: 用户录制1分钟视频演示问题（屏幕录制+语音讲解），AI自动转换成文字需求+截图标注+关键帧提取。
   🌟 **启发点**: 视频是否比文字更高效？如何降低非文字用户的门槛？

**检测算法（简化版）**:
```javascript
function detectQuality(text) {
  let score = 0;
  let missing = [];
  
  // 1. 字数检测 (20分)
  if (text.length >= 150) score += 20;
  else if (text.length >= 80) score += 10;
  else missing.push("字数不足");
  
  // 2. 背景检测 (20分)
  const bgKeywords = ["当前", "现在", "目前", "现状", "以前"];
  if (bgKeywords.some(kw => text.includes(kw))) score += 20;
  else missing.push("缺少背景描述");
  
  // 3. 目标检测 (20分)
  const goalKeywords = ["希望", "期望", "需要", "想要", "实现"];
  if (goalKeywords.some(kw => text.includes(kw))) score += 20;
  else missing.push("缺少期望目标");
  
  // 4. 场景检测 (20分)
  const sceneKeywords = ["用户", "客户", "部门", "人员", "场景"];
  if (sceneKeywords.some(kw => text.includes(kw))) score += 20;
  else missing.push("缺少使用场景");
  
  // 5. 验收检测 (20分)
  const acceptKeywords = ["达到", "完成", "标准", "要求", "指标"];
  if (acceptKeywords.some(kw => text.includes(kw))) score += 20;
  else missing.push("缺少验收标准");
  
  return { score, missing };
}
```

  - 不强制拦截，保持用户自主权
