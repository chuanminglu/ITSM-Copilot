# ITSM系统需求提交优化 - 用户旅程分析报告

> **📌 报告说明**: 本报告基于MVP功能定义和需求痛点，构建典型用户画像、同理心地图和用户旅程地图，识别关键痛点时刻和优化机会，为智能体功能设计提供用户视角的依据。
>
> **分析日期**: 2025年12月8日
> **目标场景**: 广东电信ITSM系统需求提交全流程

## 第1章：用户画像（Persona）

### 👤 用户画像1：李经理（业务部门主管）

#### 基本信息

| 属性 | 描述 |
|------|------|
| **姓名** | 李建国 |
| **年龄** | 35岁 |
| **职位** | 市场营销部主管 |
| **部门** | 市场营销部 |
| **IT熟悉度** | ⭐ 低（只会用基本办公软件） |

#### 行为特征

- 📊 **使用频率**: 每周1-2次（发现业务问题时提需求）
- ⏰ **典型场景**: 工作日上午9-11点，发现业务流程问题或客户投诉后，需要立即提交系统优化需求
- 🔧 **常用功能**: 需求提交、进度查询、需求追踪
- 🎯 **工作压力**: 需要快速响应业务问题，不能因为"提不好需求"而延误解决时间

#### 核心诉求

> "我只想快速准确地把需求提交上去，不要老是被IT部门退回来让我补充信息。我不懂技术，也不知道该选哪个系统，能不能系统帮我判断一下？"

#### 主要挫折

> "每次提需求都像考试一样，30多个系统看得眼花缭乱，选错了会被退回；需求描述写简单了，IT部门又说不清楚。我只是想解决业务问题，为什么这么难？"

#### 典型行为模式

- ✅ **积极行为**: 发现问题会主动提需求，愿意配合改进
- ⚠️ **消极行为**: 被退回2次后会产生畏难情绪，可能电话沟通代替提单
- 🤔 **困惑时刻**: 不知道该选哪个系统，不知道需求该写多详细

### 🧠 同理心地图：李经理（业务部门主管）

#### 📢 SAYS（用户说了什么）

- "系统太多了，我不知道该选哪个。"
- "我写得很清楚了啊，为什么还要我补充？"
- "能不能给个模板，我照着填就行。"
- "IT部门总是退回我的需求，太浪费时间了。"
- "我只是想解决业务问题，不想学怎么写需求。"

#### 💭 THINKS（用户在想什么）

- *我是不是太笨了，连提需求都不会？*（自我怀疑）
- *IT部门是不是故意刁难我，这么简单的问题还要问这么多？*（委屈）
- *下次直接打电话给IT部门，不用这个系统了。*（逃避心理）
- *万一选错系统，会不会被领导批评？*（焦虑）
- *如果能有个智能助手帮我，该多好。*（期待）

#### 👁️ DOES（用户做了什么）

- 打开ITSM系统，看到复杂的表单后犹豫2-3分钟
- 在系统下拉框中反复滚动，寻找可能的系统名称
- 随便选一个"听起来像"的系统（靠猜测）
- 在需求描述框输入1-2句话后直接点击提交
- 提交后焦虑等待，担心被退回
- 接到IT部门电话后，需要再次解释需求背景（重复沟通）
- 被退回2次后，选择打电话而不是用系统

#### ❤️ FEELS（用户感受如何）

- 😟 **困惑**：看到30+个系统选项时，不知道选哪个
- 😫 **挫败**：提交的需求被退回时，觉得自己做错了
- 😤 **沮丧**：反复澄清同一个问题，感觉在浪费时间
- 😰 **焦虑**：担心选错系统或写错描述影响业务问题解决
- 😞 **无助**：不知道该如何写"好需求"，缺乏指导

#### 💡 洞察提炼

**核心洞察**：李经理的痛点不是"不愿意写需求"，而是"不知道怎么写需求"。他需要的不是培训，而是**即时的、场景化的引导和帮助**。

**设计启示**：

1. **降低认知负担**：30个系统选项 → 智能推荐Top 3 + 分类导航
2. **提供脚手架**：空白文本框 → 结构化模板 + 引导问题
3. **即时反馈**：提交后被退回 → 提交前实时检测 + 缺失提示
4. **建立信心**：消除"选错系统"的焦虑 → 提供信心度反馈 + "不确定"选项

#### 💭 THINKS（用户在想什么）

- *背景、目标、场景、验收标准……这些都是什么鬼？*（专业术语障碍）
- *我只想说"查询很慢"，为什么要写这么多？*（抵触情绪）
- *填这个表格太耽误时间，还不如打电话快。*（效率焦虑）
- *我是不是写错了，会不会被IT部门笑话？*（自尊心担忧）

#### 👁️ DOES（用户做了什么）

- 直接在需求描述框输入口语化问题（如"查询很慢"）
- 跳过非必填字段，只填标题和描述
- 点击提交前没有检查，直接提交
- 接到IT部门电话时，现场口头补充信息
- 下次遇到类似问题，选择私聊IT同事而不是提单

#### ❤️ FEELS（用户感受如何）

- 😟 **困惑**：不理解"背景""目标""验收标准"这些术语
- ⏰ **急躁**：填表时间太长，想快点提交回去工作
- 😣 **无奈**：被要求补充信息时，觉得很麻烦
- 😔 **挫败**：不知道自己哪里写得不够好
- 😌 **期待**：希望有简单的模板或示例

#### 💡 洞察提炼

**核心洞察**：小王的痛点是"专业术语障碍"和"效率焦虑"。她需要的不是"完整的需求工程培训"，而是**通俗易懂的提示和快速填写的方法**。

**设计启示**：

1. **通俗化术语**：不说"验收标准"，而说"怎么判断问题解决了？"
2. **降低门槛**：提供"Bug反馈""流程优化"等场景化模板
3. **提供示例**：每个模板配2个案例，用户可以"抄作业"
4. **优化体验**：目标3分钟完成填写，不影响日常工作

### 📊 设计启示

基于用户旅程分析，得出以下关键设计启示：

#### 1. **预防式而非补救式**（核心理念）

**洞察**：用户的最大痛点不是"提交后被退回"，而是"不知道需求写得不好"。

**设计原则**：

- ✅ **提交前检测** > 提交后审核（F1-F3实时检测+警告）
- ✅ **边写边提示** > 一次性提示（F1实时高亮缺失项）
- ✅ **主动引导** > 被动等待（F4自动推荐模板）

**预期效果**：

- 需求退回率从40%降至10%
- 需求澄清轮次从平均2.5次降至1次

---

#### 2. **降低认知负担**（关键策略）

**洞察**：用户的认知负担来自两方面：①系统选择的复杂性 ②需求描述的专业性

**设计原则**：

- ✅ **分类 > 列表**（F7：30+系统 → 6大类，每类5个系统）
- ✅ **搜索 > 浏览**（F8：输入关键词，1秒定位）
- ✅ **模板 > 空白框**（F5：结构化字段 vs 自由文本）
- ✅ **示例 > 说明**（F6：2个案例 vs 1000字指南）

**预期效果**：

- 系统选择时间从2分钟降至30秒
- 需求填写时间从15分钟降至5分钟

---

#### 3. **即时反馈建立信心**（体验优化）

**洞察**：用户的焦虑来自"不确定性"——不知道选对了吗、不知道写好了吗。

**设计原则**：

- ✅ **可视化质量**（F2：质量评分0-100分+星级）
- ✅ **明确标准**（F3：质量<60分=不合格，有明确阈值）
- ✅ **及时反馈**（F1：每输入50字，实时刷新检测结果）
- ✅ **信心度询问**（F9：让用户表达不确定，而非强迫选择）

**预期效果**：

- 用户提交前信心度从50%提升至85%
- 提交后焦虑等待情绪显著改善

---

#### 4. **通俗化专业术语**（用户友好）

**洞察**：业务人员看到"背景""目标""验收标准"等术语时会困惑和抵触。

**设计原则**：

| ❌ 专业术语 | ✅ 通俗表达 | 示例 |
|-----------|-----------|------|
| 需求背景 | 为什么要做这个？遇到了什么问题？ | "客户投诉查询太慢" |
| 需求目标 | 做了之后要达到什么效果？ | "查询速度从5秒降至1秒" |
| 使用场景 | 谁在什么时候会用到？ | "客服每天处理100+查询" |
| 验收标准 | 怎么判断问题解决了？ | "P95延迟<1秒" |

**实施方式**：

- F5结构化引导的字段标签使用通俗表达
- 每个字段下方有灰色提示文字（placeholder）
- 点击"?"图标，展示详细解释和示例

---

#### 5. **3分钟快速提交**（效率优先）

**洞察**：一线业务员（如小王）的核心诉求是"快速提交，不耽误工作"。

**设计原则**：

- ✅ **智能默认值**（F4：自动推荐模板，无需手动选择）
- ✅ **分步填写**（先填核心信息，高级选项可折叠）
- ✅ **快捷操作**（F6：点击示例，一键复制到描述框）
- ✅ **保存草稿**（P1功能：自动保存，下次可继续）

**预期效果**：

- Bug反馈场景：填写时间从10分钟降至3分钟
- 功能优化场景：填写时间从20分钟降至8分钟

---

#### 6. **组合功能发挥价值**（系统思维）

**洞察**：单个功能价值有限，组合使用才能产生质变效果。

**功能组合策略**：

```
┌─────────────────────────────────────────────────────────┐
│  P1（系统选择错误）解决方案组合                           │
├─────────────────────────────────────────────────────────┤

│  ↓                                                        │
│  降低选择负担 → 快速定位 → 识别不确定 → 提供帮助          │
│  （30系统→6类） （2分钟→30秒） （消除焦虑） （兜底方案）   │
```

**协同效应**：

- F4+F5+F6：模板推荐+结构化引导+示例库 = 完整的脚手架体系

#### 7. **渐进式教育用户**（长期价值）

**洞察**：用户不是"不会写需求"，而是"没有被教过怎么写需求"。

**设计原则**：

- ✅ **边用边学**（F1提示缺失项 = 告诉用户"需求应该包含什么"）
- ✅ **示例教学**（F6高质量案例 = 最好的教程）
- ✅ **反馈强化**（F2质量评分 = 正向激励，写得好就高分）
- ✅ **非强制性**（P1功能F14需求编写技巧卡片 = 可选学习）

**预期效果**：

- 3个月后，即使没有插件辅助，用户也能写出质量更高的需求
- 从"依赖工具"到"提升能力"

### 核心发现

1. **痛点定位精准**：
   - P1（系统选择错误）的根因 = 认知负担过高 + 缺少智能推荐
   - P2（需求描述简单）的根因 = 缺少结构化脚手架 + 专业术语障碍

2. **关键时刻识别**：
   - MOT1（阶段3-系统选择）：情绪低谷，需要F7+F8+F9组合救援
   - MOT2（阶段8-被退回）：体验最差，需要F1-F6提前预防
   - MOT3（阶段6-提交前）：焦虑时刻，需要F2+F3建立信心

3. **用户分层洞察**：
   - 李经理（主管）：关注"不被退回" + "快速通过"
   - 小王（一线）：关注"快速提交" + "不要太复杂"
   - 共同诉求：不要让我思考太多，系统帮我做决策

4. **MVP功能覆盖度**：
   - P0功能（F1-F9）覆盖旅程关键阶段2-6（5个阶段）
   - 功能介入时机精准，在痛点发生前提供帮助
   - 组合效应明显，预期减少40%退回率+30%选错率
