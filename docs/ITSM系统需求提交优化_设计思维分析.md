# ITSM系统需求提交优化 - 设计思维分析

## 1️⃣ 需求理解与问题重述

**原始需求**:
广东省电信ITSM系统中，用户在提交新需求时存在两个痛点：

1. 业务部门提新需求时要选择相关系统，因为系统很多，经常选错系统，导致IT支持部门的同事分派需求时产生困扰
2. 业务部门需求目标与需求描述太简单，可能只有一句话需求，导致接收需求的部门在需求澄清与需求理解方面花费大量的时间

**技术约束**:

- 使用浏览器插件实现
- 尽量不要有后端服务

**我的理解**:
这是一个关于提升需求提交质量和效率的问题。核心矛盾在于：业务人员缺乏技术背景和需求工程知识，导致需求提交不准确、不完整，最终造成IT部门的工作效率低下和沟通成本高昂。技术约束要求解决方案必须轻量化、前端化。

**澄清问题**:

1. 目前系统数量大约有多少个？业务人员平均需要从多少个系统中选择？
2. 一句话需求的典型案例是什么样的？最常见的缺失信息是哪些（背景、目标、验收标准等）？
3. 当前需求澄清平均花费多长时间？是否有量化的成本数据（如平均往返沟通次数）？
4. 业务人员的IT素养如何？是否接受过需求工程培训？
5. 是否有历史需求数据可用于训练AI模型？

## 2️⃣ 共情分析 (Empathize)

### 用户画像

**主要用户1：业务部门需求提出者**

- 角色：业务经理、产品经理、运营人员
- 技能背景：熟悉业务流程，但技术知识有限
- 工作场景：日常工作中发现问题，需要快速提交需求
- 时间压力：希望快速提交，不想花太多时间在"填表"上
- 心理状态：担心选错系统被IT部门"嫌弃"，不确定如何描述需求才算"完整"

**主要用户2：IT支持部门需求接收者**

- 角色：需求分析师、项目经理、技术主管
- 技能背景：技术能力强，但不熟悉所有业务细节
- 工作场景：需要快速理解需求、分派给正确的开发团队
- 时间压力：需求量大，希望减少澄清环节
- 心理状态：对"一句话需求"感到沮丧，频繁的需求分派错误影响团队效率

### 用户旅程

**当前业务人员提需求流程**:

1. **发现问题** → 在业务场景中遇到痛点或新需求
2. **打开ITSM** → 登录系统，找到需求提交入口
3. **😰 选择系统** → 面对众多系统列表，不确定选哪个（焦虑点）
   - 可能基于模糊记忆选择
   - 可能询问同事（打断他人工作）
   - 可能随便选一个（导致后续问题）
4. **😓 填写需求** → 只填写一句话描述（认知负担点）
   - "不知道该写什么"
   - "写太多也不一定有人看"
   - "反正后面会有人来问我"
5. **提交** → 完成提交，等待反馈
6. **😤 被退回/反复沟通** → IT部门要求补充信息或重新分派（挫败点）

**当前IT人员处理需求流程**:

1. **接收需求** → 收到新需求通知
2. **😒 阅读一句话需求** → 信息不足，无法判断（沮丧点）
3. **😤 发现系统选错** → 需要重新分派（浪费时间）
4. **📞 反复沟通澄清** → 通过邮件、电话、会议追问细节（高成本）
5. **重新整理需求** → 自己补充需求文档（额外工作量）
6. **分派给开发团队** → 终于可以开始真正的工作

### 痛点识别

| 表面痛点               | 深层痛点                                                                         |
| ---------------------- | -------------------------------------------------------------------------------- |
| 系统太多，不知道选哪个 | **认知负载过高**：业务人员缺乏系统架构知识，无法将业务问题映射到技术系统   |
| 需求描述只有一句话     | **需求工程能力缺失**：业务人员不知道"什么是好需求"，没有结构化思维框架     |
| IT部门分派需求困扰     | **信息不对称**：业务人员知道问题但不知道如何表达，IT人员有能力但缺少上下文 |
| 需求澄清耗时长         | **异步沟通低效**：需求提交后才发现信息不足，导致多轮往返                   |

### 同理心洞察

| 用户说的                       | 用户做的             | 用户想的                         | 用户感受的     |
| ------------------------------ | -------------------- | -------------------------------- | -------------- |
| "系统太多了，不知道选哪个"     | 随便选一个或问同事   | "选错了会不会被批评？"           | 焦虑、不自信   |
| "需求很简单，一句话就能说清楚" | 只填写标题或一行描述 | "写多了也没人看，后面会有人问我" | 敷衍、无奈     |
| "IT部门总是来回问问题"         | 被动等待IT部门联系   | "我已经说了，为什么还不明白？"   | 沮丧、被误解   |
| "需求提交流程太复杂"           | 抗拒详细填写表单     | "我很忙，没时间填这么多"         | 抵触、时间压力 |

> 💡 **核心洞察**:
>
> 这不是一个"用户偷懒"的问题，而是**认知鸿沟**问题。业务人员处于"不知道自己不知道什么"的状态（Unconscious Incompetence）。他们：
>
> - 不知道需求需要包含哪些要素
> - 不知道如何将业务问题翻译成技术语言
> - 不知道自己的描述对IT人员来说信息严重不足
>
> 传统的解决方案是"培训业务人员"或"强制填写必填项"，但这会增加业务人员的认知负担和抵触情绪。
>
> **真正需要的是**：一个"智能翻译官" + "思维脚手架"，帮助业务人员**在不增加负担的前提下**，自然而然地提供完整信息。

## 3️⃣ 问题定义 (Define)

### 问题重构 (HMW)

**HMW 1**: 我们如何帮助**业务人员**在**提交需求时**准确选择相关系统，**即使他们不了解系统架构**？

**HMW 2**: 我们如何帮助**业务人员**在**描述需求时**提供完整信息，**而不让他们觉得是在填写复杂表单**？

**HMW 3**: 我们如何让**需求提交过程**成为一个**引导式对话**，而非**填表任务**？

### POV 陈述 (Point of View)

**业务人员**需要**一个智能助手**，因为**他们有清晰的业务问题，但缺乏将问题转化为结构化需求的能力和知识框架**。

**IT支持人员**需要**结构化、完整的需求信息**，因为**只有这样才能准确理解需求、快速分派、减少返工**。

### 核心矛盾

1. **效率 vs 质量**: 业务人员希望快速提交（效率优先），IT人员希望信息完整（质量优先）
2. **自由表达 vs 结构化**: 业务人员习惯自然语言描述，IT人员需要结构化信息
3. **减少负担 vs 增加信息**: 不能简单地"增加必填项"，否则会增加业务人员抵触
4. **前端实现 vs 智能能力**: 技术约束要求前端化，但智能推荐通常需要后端模型

### 优先级判断

**这是一个"止血+增值"问题**：

- **止血**：当前的错误分派和反复澄清造成明确的成本浪费（止血优先级高）
- **增值**：提升需求质量可以加速后续开发流程，提升整体交付效率（长期价值）

**建议策略**：先用MVP快速止血（解决系统选择问题），再逐步优化需求描述质量。

## 5️⃣ 原型与验证 (Prototype & Test)

### 🎯 推荐方案：**混合策略 - 分阶段实施**

**阶段1（立即可做，0-2周）**：方案E零代码 + 方案C模板

- 快速上线静态检查清单工具
- 同时设计3-5个需求模板
- 通过邮件/培训推广使用

**阶段2（MVP，2-4周）**：方案B历史数据智能推荐插件

- 开发浏览器插件，注入ITSM页面
- 基于历史数据实现系统推荐（关键词匹配）
- 基于模板提示需求完整性

**阶段3（优化，4-12周）**：方案A AI对话式引导（可选）

- 如果浏览器AI能力成熟，升级为对话式
- 如果预算允许，可接入轻量后端API

### MVP 定义（方案B为核心）

**MVP功能范围**:

✅ **必须有**：

1. **系统智能推荐**：

   - 用户输入需求描述后，插件分析关键词
   - 推荐Top 3相关系统，显示推荐理由（如"包含关键词：计费、账单"）
   - 用户可一键选择或手动选择其他系统
2. **需求完整性检查**：

   - 基于预设规则检查需求描述（如字数<50字提示"描述过于简单"）
   - 展示需求模板提示（如"功能优化类需求通常包含：当前问题、期望效果、使用场景"）
   - 高亮缺失的关键信息（如"未提及业务背景"）
3. **一键优化建议**：

   - 基于选择的需求类型，生成补充问题（如"这个问题影响多少用户？"）
   - 用户点击问题后，在需求描述框插入提示文本

⚠️ **暂不包含**：

- 复杂的AI对话交互
- 后端API调用
- 历史需求搜索（留待阶段3）

**技术实现方案**：

```
浏览器插件架构：
├── Content Script（注入ITSM页面）
│   ├── 监听需求描述输入框
│   ├── 提取用户输入文本
│   └── 注入推荐UI组件
├── Background Script（后台逻辑）
│   ├── 关键词提取（TF-IDF）
│   ├── 系统匹配规则引擎
│   └── 模板库管理
├── 数据文件（打包在插件中）
│   ├── system_keywords.json（系统-关键词映射）
│   ├── templates.json（需求模板库）
│   └── rules.json（完整性检查规则）
└── UI组件
    ├── 推荐卡片（系统推荐）
    ├── 提示面板（完整性检查）
    └── 模板选择器
```

**数据准备**（需人工处理）：

1. 导出历史需求数据（需求描述 + 最终分派的系统）
2. 人工标注50-100条典型案例的关键词
3. 提取系统-关键词映射规则
4. 设计3-5个需求模板

### 验证方法

**定量指标**（A/B测试）：

| 指标           | 当前基线   | 目标值            | 测量方法                                         |
| -------------- | ---------- | ----------------- | ------------------------------------------------ |
| 系统选择准确率 | （需测量） | 提升30%           | 对比使用插件前后，需求首次分派的正确率           |
| 需求澄清轮次   | （需测量） | 减少50%           | 统计IT部门联系业务人员的平均次数                 |
| 需求描述完整度 | （需测量） | 80%需求包含5W要素 | 人工抽查需求描述，检查是否包含背景、目标、场景等 |
| 需求提交时长   | （需测量） | 不增加或增加<30秒 | 记录用户从打开表单到提交的时间                   |

**定性验证**（用户访谈）：

**业务人员访谈问题**：

1. 插件推荐的系统是否符合你的预期？
2. 需求完整性提示是否有帮助？是否感到"被强迫"？
3. 使用插件后，提需求是否更有信心？
4. 你会推荐同事使用吗？

**IT人员访谈问题**：

1. 使用插件后，收到的需求质量是否有提升？
2. 错误分派的情况是否减少？
3. 澄清需求的时间是否减少？
4. 还有哪些信息经常缺失？

**快速验证原型**（2周内可完成）：

**原型1：系统推荐规则验证**

- 工具：Excel + 简单Python脚本
- 方法：
  1. 导出50条历史需求
  2. 基于关键词匹配规则预测系统
  3. 对比实际分派结果
  4. 计算准确率，调整规则
- 成功标准：准确率>70%

**原型2：需求模板可用性测试**

- 工具：Google Docs + 模板
- 方法：
  1. 邀请5名业务人员
  2. 给出一个真实场景，要求用模板填写需求
  3. 观察是否能顺利完成，记录困惑点
  4. 访谈模板是否易用

## 💡 额外洞察与建议

### 从"填表"到"对话"的范式转变

这个需求的本质是**降低需求工程的门槛**，让不懂需求工程的人也能提交高质量需求。

传统思路是"培训用户"或"强制填写必填项"，但这治标不治本。真正的解决方案是**范式转变**：

| 传统范式（表单思维） | 新范式（对话思维） |
| -------------------- | ------------------ |
| 用户填写表单         | AI引导对话         |
| 必填项强制要求       | 智能提问补充信息   |
| 错误后反馈           | 实时预防错误       |
| 用户需要学习规则     | 系统适应用户习惯   |

**建议**：即使当前技术约束限制了AI能力，也应该在UI设计上体现"对话"思维，而非传统的"表单"思维。例如：

- 不要一次展示所有字段，而是逐步展开
- 用自然语言提示代替专业术语（"这个需求会影响多少用户？"而非"影响范围"）
- 用案例引导而非抽象说明

### 从"工具"到"教育"的双重价值

插件不仅是一个效率工具，更是一个**需求工程教育工具**。

通过长期使用，业务人员会：

- 潜移默化地学习"什么是好需求"
- 建立结构化思维习惯
- 理解IT部门的工作方式

**建议**：在插件中嵌入"知识卡片"，例如：

- "为什么需要描述使用场景？"→ 帮助开发者理解需求背景
- "为什么需要说明预期效果？"→ 用于后续验收测试

这样插件不仅解决当前问题，还能**提升团队整体的需求工程能力**。

### 警惕"过度优化"陷阱

虽然AI对话式方案（方案A）用户体验最好,但要警惕"过度工程化"：

- 如果历史数据质量差，AI也无法准确推荐
- 如果业务人员抗拒使用，再智能的工具也无用
- 如果IT部门流程本身有问题，工具只能缓解症状

**建议**：先用MVP快速验证核心假设（系统推荐是否有效？模板是否有帮助？），再决定是否投入更多资源开发高级功能。
